computeRooflineCoExec() - Interactive Explanation

Purpose: This method analyzes a scheduling region before scheduling to calculate the theoretical maximum instruction hiding (co-execution) achievable. It uses a max-flow algorithm to optimally match consumer instructions to available co-execution slots.

Why "Roofline"? Like roofline models for memory/compute bounds, this establishes an upper bound on performance — the best possible slot utilization given the instruction mix.

1. Understanding Co-Execution Slots

WMMA (Wave Matrix Multiply-Accumulate) instructions take many cycles. During execution, other instructions can execute in parallel in co-execution slots.

WMMA Stage Pattern Example: 0EIIEEIIEEIIEEIIV

This 17-cycle pattern for V_WMMA_16x16x64_IU8 shows which instructions can co-execute at each cycle:

E0 (CTRL only) - skipped in roofline
E (SALU, MEM)
I (SALU, MEM, VALU, TRANS)
V (SALU, MEM, next WMMA)
IS (I + next scaled WMMA)

CoExecMask Bit Definitions

namespace CoExecMask {
  CTRL  = 1 << 0  // Control instructions (s_delay_alu)
  VALU  = 1 << 1  // Vector ALU
  TRANS = 1 << 2  // Transcendentals (v_exp, v_log)
  SALU  = 1 << 3  // Scalar ALU
  DS    = 1 << 4  // LDS read/write
  VMEM  = 1 << 5  // Global memory
  SMEM  = 1 << 6  // Scalar memory
  WMMA  = 1 << 7  // Next WMMA instruction

  StageE0 = CTRL                    // 0x01
  StageE  = CTRL | SALU | MEM       // 0x79
  StageI  = CTRL | SALU | MEM | VALU | TRANS  // 0x7F
  StageV  = CTRL | SALU | MEM | WMMA // 0xF9
}

2. Algorithm Overview

Step 1: Count Slot Types
Scan all WMMAs, enumerate their stage patterns, collect slots into a map keyed by mask. Skip E0 stages (filled by hazard recognizer).
Step 2: Count Consumers
Classify all non-WMMA instructions by flavor (VALU, TRANS, SALU, DS, VMEM, SMEM) and count each type.
Step 3: Build Flow Network
Construct a bipartite graph: Source → Consumer Classes → Slot Types → Sink. Edge capacities reflect counts and compatibility.
Step 4: Solve Max-Flow
Run Edmonds-Karp (BFS-based Ford-Fulkerson) to find maximum matching of consumers to slots.
Step 5: Compute Results
MaxFilledSlots = flow value. LowerBoundStalls = TotalSlots - MaxFilledSlots. ExposedByClass = unfilled per consumer type.

5. TinyMaxFlow Implementation

The algorithm uses a compact Edmonds-Karp implementation (BFS-based Ford-Fulkerson):

Data Structure

struct TinyMaxFlow {
  static constexpr unsigned MaxNodes = 20;
  int Capacity[MaxNodes][MaxNodes];   // Residual capacity graph
  int Original[MaxNodes][MaxNodes];   // Snapshot for flow recovery
  unsigned NumNodes = 0;
  int Parent[MaxNodes];               // BFS parent pointers

  void init(unsigned N) {
    NumNodes = N;
    memset(Capacity, 0, sizeof(Capacity));
    memset(Original, 0, sizeof(Original));
  }

  void addEdge(unsigned From, unsigned To, int Cap) {
    Capacity[From][To] += Cap;
    Original[From][To] += Cap;
  }
};

findAugmentingPath() - BFS from Source to Sink

// Returns true if a path exists from Source (0) to Sink (NumNodes-1)
// Populates Parent[] array to reconstruct the path
bool findAugmentingPath() {
  unsigned Source = 0;
  unsigned Sink = NumNodes - 1;

  // Initialize parent pointers to -1 (unvisited)
  memset(Parent, -1, sizeof(Parent));
  Parent[Source] = Source;  // Mark source as visited

  // BFS queue
  std::queue<unsigned> Q;
  Q.push(Source);

  while (!Q.empty() && Parent[Sink] == -1) {
    unsigned U = Q.front();
    Q.pop();

    // Explore all neighbors with positive residual capacity
    for (unsigned V = 0; V < NumNodes; ++V) {
      if (Parent[V] == -1 && Capacity[U][V] > 0) {
        Parent[V] = U;      // Record how we reached V
        Q.push(V);
      }
    }
  }

  return Parent[Sink] != -1;  // Did we reach the sink?
}

findBottleneck() - Minimum capacity along the path

// Walk backward from Sink to Source using Parent[]
// Return the minimum edge capacity (the bottleneck)
int findBottleneck() {
  unsigned Source = 0;
  unsigned Sink = NumNodes - 1;
  int PathFlow = std::numeric_limits<int>::max();

  // Traverse path backward: Sink ← ... ← Source
  for (unsigned V = Sink; V != Source; V = Parent[V]) {
    unsigned U = Parent[V];
    PathFlow = std::min(PathFlow, Capacity[U][V]);
  }

  return PathFlow;
}

updateResidual() - Push flow and update capacities

// Push pathFlow units along the path found by BFS
// Key insight: decrease forward capacity, INCREASE reverse capacity
// The reverse capacity enables "undoing" flow in future iterations
void updateResidual(int PathFlow) {
  unsigned Source = 0;
  unsigned Sink = NumNodes - 1;

  for (unsigned V = Sink; V != Source; V = Parent[V]) {
    unsigned U = Parent[V];
    Capacity[U][V] -= PathFlow;  // Reduce forward capacity
    Capacity[V][U] += PathFlow;  // Add reverse capacity (for backflow)
  }
}

Complete solve() and getFlow()

int solve() {
  unsigned Source = 0;
  unsigned Sink = NumNodes - 1;
  int TotalFlow = 0;

  // Repeat: find path, push flow, until no path exists
  while (findAugmentingPath()) {
    int PathFlow = findBottleneck();
    updateResidual(PathFlow);
    TotalFlow += PathFlow;
  }

  return TotalFlow;
}

// Recover per-edge flow after solve():
// Flow = (original capacity) - (remaining capacity)
int getFlow(unsigned From, unsigned To) const {
  return Original[From][To] - Capacity[From][To];
}

6. Key Insights

Why Max-Flow?

Greedy assignment fails when slot types overlap. E.g., TRANS can go in I-slots, but so can VALU. Max-flow finds the globally optimal assignment.

E0 Skipped

E0 slots only accept CTRL instructions (s_delay_alu). The hazard recognizer inserts these, so roofline excludes them from fillable slots.

WMMA as Consumer

With N WMMAs, N-1 can fill V-slots of preceding WMMAs. This is why ConsumerCount[WMMA] += NumWMMAs - 1.

ExposedByClass

Instructions that couldn't be placed in any slot are "exposed" — they add to critical path and can't be hidden.

7. How Scheduling Uses This

The roofline provides:

  • LowerBoundStalls: Minimum unavoidable stalls (unfillable slots)
  • ExposedByClass: Which instruction types are bottlenecked
  • Heuristic guidance: Prioritize scheduling exposed classes that match available slots

If VALU is exposed but SALU is not, the scheduler knows to prioritize VALU instructions for I-slots over SALU (which can also use E-slots).
